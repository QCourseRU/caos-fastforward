# Сигналы и пайпы!

## 1. Что такое сигнал

Сигнал — одно число от 1 до 64. Ядро
или процесс говорит процессу «случилось событие N». Имена `SIGINT`, `SIGSEGV` — просто `#define` поверх этих чисел.

### Таксономия

Сигналы делятся на несколько осей, и это важно понимать сразу:

**Синхронные vs асинхронные.**
- *Синхронные* генерируются самим потоком в результате его действий:
  `SIGSEGV`, `SIGBUS`, `SIGFPE`, `SIGILL`, `SIGTRAP`. Ядро может точно
  указать адрес инструкции, которая их вызвала. Они попадают ровно
  в тот тред, который «согрешил».
- *Асинхронные* — результат внешнего события: `SIGINT`, `SIGTERM`,
  `SIGUSR1`, `SIGALRM`, `SIGCHLD`. Приходят в произвольный момент,
  не привязанный к коду процесса.

**Стандартные vs real-time.**
- Стандартные (номера 1..31): битовая маска, дубли при блокировке
  теряются!
- Real-time (32..64, псевдонимы `SIGRTMIN`..`SIGRTMAX`): очередь,
  все доставятся в порядке отправки, у каждого может быть свой
  `sigval`.

**По семантике дефолтного действия:** `Term`, `Core`, `Ign`, `Stop`,
`Cont`. Полная таблица — `man 7 signal`.

### Жизненный цикл сигнала

```
  Источник (kill / ядро / TTY)
     │
     ▼
  __send_signal():
     - для стандартных: бит в sigset_t (дубли теряются)
     - для RT: sigqueue_t в списке
     │
     ▼
  TIF_SIGPENDING у целевого треда + wake_up()
     │
     ▼
  Тред выходит из ядра в user-space (после syscall / прерывания)
     │
     ▼
  do_signal() → dequeue_signal() → sigaction
     │
     ▼
  либо SIG_DFL / SIG_IGN / userspace handler (signal frame)
```

**Ключевая идея:** сигнал не может быть доставлен в произвольный момент
в user-space. Доставка происходит только при возврате из ядра —
после завершения syscall или обработки прерывания. Пока тред крутится
в user-space без системных вызовов, сигнал «висит» в `pending`.

> Если бесконечный цикл `while(1);` в user-space
> ловит `SIGTERM`, то сработает handler тогда, когда ядро
> вытеснит процесс по таймерному прерыванию и будет возвращаться —
> попутно доставит сигнал!

## 2. Обработчики: `signal` vs `sigaction`

### Минимальный пример

```c
#include <signal.h>
#include <unistd.h>

static void sayhi(int signo) {
    (void)signo;
    write(STDOUT_FILENO, "hi\n", 3);
}

int main(void) {
    signal(SIGINT, sayhi);
    for (;;) pause();
}
```

### `EINTR` и `SA_RESTART`

Когда сигнал приходит во время блокирующего системного вызова
(`read`, `write`, `accept`, `wait`, …), возможны два исхода:

1. **Без `SA_RESTART`:** syscall возвращает `-1`, `errno = EINTR`.
   Программа сама должна решать, что делать.
2. **С `SA_RESTART`:** ядро само перезапускает syscall после handler'а.

Однако `SA_RESTART` работает **не для всех syscall'ов**. Например,
`select`, `poll`, `epoll_wait`, `connect`, `recv` с таймаутом,
`semop`, `read` на сокете/терминале с таймером — всегда возвращают
`EINTR`, независимо от флага. Полный список — в `man 7 signal`
(секция «Interruption of system calls and library functions by signal
handlers»).

**Работа с `EINTR`:**

```c
ssize_t safe_read(int fd, void* buf, size_t n) {
    for (;;) {
        ssize_t r = read(fd, buf, n);
        if (r < 0 && errno == EINTR) {
          continue;
        }
        return r;
    }
}
```

## 3. Что со обработкой?

Для выполнения функции sayhi необходимо иметь стек. В любой момент времени может прийти сигнал, и нужно будет одновременно выполнять main и sayhi. Ядро копирует все состояние программы и запускает sayhi на стеке ниже esp, как обычную функцию с точки зрения копии. Однако оригинальный процесс приостанавливается, и после завершения обработчика возобновляется.

## 4. Маски, `pending`/`blocked` и гонка `pause()`

Для каждой пары (тред, сигнал) ядро помнит:

- `pending` — сигнал ожидает доставки (пришёл, но ещё не выполнен).
- `blocked` — сигнал заблокирован маской, не может быть доставлен
  сейчас.

**Для стандартных сигналов pending — это один бит.** Поэтому если
сигнал заблокирован, и пришёл два раза — это эквивалентно одному разу.
Для RT-сигналов это очередь.

### Классическая гонка

Мы хотим ждать «события», которое отмечается флагом в handler'е:

```c
volatile sig_atomic_t got = 0;
void h(int s) {
  got = 1;
}

/* ... */
while (!got) pause();  // BUG
```

Проблема: между `while(!got)` и `pause()` может прилететь сигнал.
Handler отработает, `got=1`, но `pause()` уже вошёл в ядро и будет
ждать **следующего** сигнала — а его может никогда не быть.

### Починка: `sigsuspend`

Системный вызов `sigsuspend(&mask)` атомарно:
1. Устанавливает маску сигналов.
2. Усыпляет процесс до прихода сигнала, который не в маске.
3. По возврату восстанавливает старую маску.

Идиома:

```c
sigset_t block, old, wait;
sigemptyset(&block);
sigaddset(&block, SIGUSR1);

/* 1. блокируем сигнал на время критической секции */
sigprocmask(SIG_BLOCK, &block, &old);

/* 2. проверяем флаг и спим атомарно */
wait = old;  /* во время ожидания SIGUSR1 разблокирован */
while (!got) {
    sigsuspend(&wait);
}

sigprocmask(SIG_SETMASK, &old, NULL);
```

Ключ в том, что между проверкой `!got` и `sigsuspend` сигнал
**заблокирован** — доставиться не может. А `sigsuspend` разблокирует
ровно в тот момент, когда усыпляет тред.

## 5. `SIGCHLD` и сбор зомби

Когда дочерний процесс завершается, ядро:
1. Оставляет в таблице процессов зомби-запись (нужна для `wait`).
2. Шлёт родителю `SIGCHLD`.

**Три способа корректно обработать:**

### Способ 1: блокирующий `wait` в main'е

Если родитель в какой-то момент всё равно вызывает `wait`/`waitpid` —
ничего не надо.

### Способ 2: неблокирующий `waitpid` в handler'е `SIGCHLD`

```c
static void reap(int signo) {
    (void)signo;
    // WNOHANG - не блокироваться, если нет готовых потомков
    while (waitpid(-1, NULL, WNOHANG) > 0) { }
}

struct sigaction sa = {0};
sa.sa_handler = reap;
sa.sa_flags = SA_NOCLDSTOP | SA_RESTART;  // не триггерить на stop/cont
sigemptyset(&sa.sa_mask);
sigaction(SIGCHLD, &sa, NULL);
```

Обратите внимание на `while` — `SIGCHLD` может быть *коалесцирован*:
пока handler работает, два потомка могут умереть — мы получим только
один сигнал, но двух зомби. Поэтому `waitpid` в цикле до `0`/`-1`.

### Способ 3: `SIG_IGN` на `SIGCHLD`

```c
signal(SIGCHLD, SIG_IGN);
```

Это специальный случай — ядро автоматически подчищает потомков,
`wait` невозможен (всегда вернёт `ECHILD`). Удобно для сервисов,
которым статус потомков не нужен.

## Пайпы: базовая семантика

`pipe(int fds[2])` создаёт unidirectional канал: `fds[0]` — read end,
`fds[1]` — write end. Буфер живёт в памяти ядра, размер на Linux —
16 страниц = 64 КБ по умолчанию.

```c
int fds[2];
pipe(fds);

pid_t kid = fork();
if (kid == 0) {
    close(fds[0]); 
    write(fds[1], "hi\n", 3);
    close(fds[1]);
    _exit(0);
}
close(fds[1]); 
char buf[64];
ssize_t n = read(fds[0], buf, sizeof(buf));
```

### Матрица поведения

**`read(pipe_fd, buf, n)`:**

| состояние | поведение |
|-----------|-----------|
| есть данные | читает до `n` байт, возвращает прочитанное |
| нет данных, есть writers | блокируется до появления данных или EOF |
| нет данных, нет writers | возвращает 0 (EOF) |
| `O_NONBLOCK` + нет данных + есть writers | `-1`, `EAGAIN` |

**`write(pipe_fd, buf, n)`:**

| состояние | поведение |
|-----------|-----------|
| есть место | записывает, возвращает записанное |
| буфер полон, есть readers | блокируется до освобождения места |
| нет readers | ядро шлёт `SIGPIPE` → `write` → `-1`, `EPIPE` |
| `O_NONBLOCK` + полный + readers | `-1`, `EAGAIN` |

### Главная ловушка: забыть `close`

Если после fork'а вы не закрыли write-end в родителе, а потом
читаете из read-end — родитель **ждёт EOF, которого не будет**,
потому что сам же держит открытый write-end.

Правило: после fork'а каждая сторона **сразу** закрывает те концы,
которые ей не нужны. Включая те, что дуплицированы через `dup2`.

### `PIPE_BUF` и атомарность

POSIX гарантирует: запись `write(fd, buf, n)` при `n <= PIPE_BUF`
атомарна — никто другой не вклинится в середину. На Linux
`PIPE_BUF = 4096`.

При `n > PIPE_BUF` запись может быть разбита на несколько операций;
в промежутке другой писатель может вставить свои данные.

**Следствие:** многопроцессное логирование в один pipe/FIFO работает
корректно, если каждая запись <= 4096 байт. Для больших записей
нужен отдельный mutex или один logger-процесс.

**Следствие 2:** syslog, журналирование систем — все используют
PIPE_BUF атомарность как базовую гарантию. Не ломайте её длинными
записями.

## 14. Именованные каналы (FIFO)

`pipe(2)` работает только между родственниками. `mkfifo(3)` создаёт
имя в ФС (специальный файл типа `p`), через которое могут
общаться **любые** процессы:

```bash
$ mkfifo /tmp/myfifo
$ ls -la /tmp/myfifo
prw-r--r-- 1 user user 0 Apr 22 10:00 /tmp/myfifo
↑ «p» = named pipe
```

Тонкости:
- `open(fifo, O_RDONLY)` блокируется, пока кто-то не откроет на write.
- `open(fifo, O_WRONLY)` блокируется наоборот.
- `open(fifo, O_RDWR)` — технически работает, не блокируется, но
  семантика не стандартизирована POSIX.
- `O_NONBLOCK` на open читателя — открывает сразу. На open писателя
  без читателя — возвращает `-1`, `ENXIO`.
- Как только оба конца открыты — FIFO работает ровно как обычный pipe:
  тот же kernel-buffer, та же `PIPE_BUF`-атомарность, тот же `SIGPIPE`.

Применение: межпроцессное IPC без общего родителя, простейшие очереди,
отладочные сокеты.

## 15. Отладка сигналов и пайпов

### strace

```bash
$ strace -f -e trace=signal ./program
```

Покажет все доставленные сигналы и их источник. Крайне полезно, когда
«программа загадочно падает» — часто окажется, что ей пришёл `SIGPIPE`
или `SIGTERM` от systemd.

### /proc/\<pid\>/status

Маски `SigBlk`, `SigIgn`, `SigCgt`, `SigPnd`, `ShdPnd` показывают
состояние сигналов процесса в хекс-битмаске.

```bash
$ grep ^Sig /proc/$$/status
SigQ:   0/30526
SigPnd: 0000000000000000
SigBlk: 0000000000010000
SigIgn: 0000000000384000
SigCgt: 000000004b817efb
```

### /proc/\<pid\>/fd

Покажет, какие pipe'ы открыты у процесса:

```bash
$ ls -la /proc/$$/fd
lrwx------ ... 3 -> 'pipe:[1234567]'
```

Одинаковый inode `pipe:[1234567]` в двух разных процессах = оба
держат один конец одного пайпа.
