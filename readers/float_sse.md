# Вещественные числа и SSE

## Зачем нужен отдельный формат?

Целые числа на машине ведут себя почти как арифметика по модулю степени двойки.
Вещественные числа устроены иначе:
* не каждое десятичное или целое число представимо точно;
* после операций происходит округление;
* есть специальные значения: `+0`, `-0`, `+inf`, `-inf`, `NaN`;
* привычные свойства вроде ассоциативности сложения могут ломаться.

На современных x86-64 вещественная арифметика обычно выполняется не через старый
x87-стек, а через SSE/AVX-регистры. В рамках курса достаточно понимать SSE и
регистры `xmm`.

## Формат IEEE 754

`float` занимает 32 бита:

```
31      30           23 22                         0
+-------+--------------+---------------------------+
| sign  | exp          | frac                      |
+-------+--------------+---------------------------+
```

Поля:
* `sign` — знаковый бит;
* `exp` — показатель степени со смещением;
* `frac` — дробная часть мантиссы.

Для `float`:
* `exp` занимает 8 бит;
* `frac` занимает 23 бита;
* смещение показателя степени равно `127`.

Для `double`:
* `exp` занимает 11 бит;
* `frac` занимает 52 бита;
* смещение показателя степени равно `1023`.

## Нормализованные числа

Если `0 < exp < 255`, то `float` является нормализованным числом:

```
(-1)^sign * 2^(exp - 127) * 1.frac
```

Единица перед точкой не хранится явно, поэтому у `float` фактически 24 значащих
бита: 23 бита `frac` и скрытая единица.

Пример:

```
0x40a00000 = 0100 0000 1010 0000 0000 0000 0000 0000


0 | 10000001 | 01000000000000000000000
^   ^^^^^^^^   ^^^^^^^^^^^^^^^^^^^^^^^
sign   exp              frac
```

Это:
* `sign = 0`;
* `exp = 129`;
* настоящий показатель степени: `129 - 127 = 2`;
* мантисса: `1.25`;
* значение: `1.25 * 2^2 = 5.0`.

Проверка:

```c
#include <stdint.h>
#include <stdio.h>

int main() {
    union {
        uint32_t u;
        float f;
    } x = { .u = 0x40a00000 };

    printf("%f\n", x.f);
    printf("%a\n", x.f);
}
```

**OUTPUT:**
```
5.000000
0x1.4p+2
```

## Денормализованные числа

Если `exp == 0`, то число либо равно нулю, либо является денормализованным:

```
(-1)^sign * 2^(-126) * 0.frac
```

У денормализованных чисел нет скрытой единицы. Они нужны, чтобы числа около нуля
не обрывались резко.

Минимальное положительное нормализованное `float`:

```
2^-126
```

Минимальное положительное денормализованное `float`:

```
2^-149
```

## Нули, бесконечности и NaN

Специальные случаи:
* `exp == 0`, `frac == 0` — ноль;
* `exp == 255`, `frac == 0` — бесконечность;
* `exp == 255`, `frac != 0` — `NaN`.

У нуля есть знак:

```c
#include <stdint.h>
#include <stdio.h>

int main() {
    union {
        float f;
        uint32_t u;
    } pz = { .f = 0.0f }, nz = { .f = -0.0f };

    printf("+0 == -0: %d\n", pz.f == nz.f);
    printf("+0 bits: 0x%08x\n", pz.u);
    printf("-0 bits: 0x%08x\n", nz.u);
}
```

**OUTPUT:**
```
+0 == -0: 1
+0 bits: 0x00000000
-0 bits: 0x80000000
```

`+0.0` и `-0.0` сравниваются как равные, но могут вести себя по-разному:

```c
#include <stdio.h>

int main() {
    printf("%f\n", 1.0 / 0.0);
    printf("%f\n", 1.0 / -0.0);
}
```

**OUTPUT:**
```
inf
-inf
```

## NaN

`NaN` означает "not a number". Например, он может появиться при `0.0 / 0.0` или
`sqrt(-1.0)`.

Главное свойство: `NaN` не равен ничему, включая самого себя.

```c
#include <math.h>
#include <stdio.h>

int main() {
    double x = 0.0 / 0.0;

    printf("x == x: %d\n", x == x);
    printf("x != x: %d\n", x != x);
    printf("isnan(x): %d\n", isnan(x));
}
```

**OUTPUT:**
```
x == x: 0
x != x: 1
isnan(x): 1
```

**Вывод:** для проверки на NaN надо использовать `isnan`, а не сравнение.

## Потеря точности

У `float` только 24 значащих двоичных бита. Пока числа маленькие, этого хватает,
чтобы представлять соседние целые числа подряд. Но около `2^24` последний
значащий бит уже соответствует шагу 2, поэтому представимы `2^24`,
`2^24 + 2`, `2^24 + 4`, ... А число `2^24 + 1` представить нельзя, оно
округляется обратно к `2^24`.

```c
#include <stdio.h>

int main() {
    float a = 16777216.0f; // 2^24
    float b = a + 1.0f;
    float c = a + 2.0f;

    printf("a = %.0f\n", a);
    printf("b = %.0f\n", b);
    printf("c = %.0f\n", c);
    printf("a == b: %d\n", a == b);
    printf("a == c: %d\n", a == c);
}
```

**OUTPUT:**
```
a = 16777216
b = 16777216
c = 16777218
a == b: 1
a == c: 0
```

В районе `2^24` шаг между соседними `float` равен 2.

## Округление

Операции над вещественными числами обычно дают точный математический результат,
после чего он округляется до ближайшего представимого числа.

Стандартный режим округления — к ближайшему, при ровной середине к чётному.

Примерная идея:

```
1.010111...  -> ближайшее представимое число
1.011000...  -> если ровно середина, выбирается вариант с чётным младшим битом
```

Такой режим уменьшает систематическую ошибку при большом количестве операций.

## Неассоциативность

Для вещественных чисел сложение не ассоциативно:

```c
#include <stdio.h>

int main() {
    double a = 1e100;
    double b = -1e100;
    double c = 3.14;

    printf("(a + b) + c = %.17g\n", (a + b) + c);
    printf("a + (b + c) = %.17g\n", a + (b + c));
}
```

**OUTPUT:**
```
(a + b) + c = 3.1400000000000001
a + (b + c) = 0
```

Во втором выражении `3.14` теряется при сложении с очень большим по модулю
числом.

**Вывод:** компилятор не имеет права произвольно переставлять вещественные
операции, если ему явно не разрешили это флагами вроде `-ffast-math`.

## Суммирование массива

Обычная сумма:

```c
double sum_simple(const double *a, int n) {
    double s = 0.0;
    for (int i = 0; i < n; ++i) {
        s += a[i];
    }
    return s;
}
```

Порядок сложения влияет на результат. Часто полезно складывать числа от меньших
по модулю к большим или использовать компенсационные алгоритмы.

Алгоритм Кэхэна:

```c
double sum_kahan(const double *a, int n) {
    double sum = 0.0;
    double c = 0.0;

    for (int i = 0; i < n; ++i) {
        double y = a[i] - c;
        double t = sum + y;
        c = (t - sum) - y;
        sum = t;
    }

    return sum;
}
```

Идея: переменная `c` хранит накопленную ошибку округления, которую пробуем
вернуть в следующие итерации.

## Приведения типов в C

Полезные факты:
* `float -> double` выполняется точно;
* `double -> float` может потерять точность или стать бесконечностью;
* `int32_t -> double` выполняется точно;
* `int32_t -> float` может округлиться;
* `float/double -> int` отбрасывает дробную часть;
* `float/double -> int` не определено, если значение не помещается в целевой тип.

Пример:

```c
#include <stdio.h>

int main() {
    int x = 16777217;
    float f = x;
    int y = f;

    printf("x = %d\n", x);
    printf("f = %.0f\n", f);
    printf("y = %d\n", y);
}
```

**OUTPUT:**
```
x = 16777217
f = 16777216
y = 16777216
```

## Старый x87 и SSE

Исторически на x86 для вещественной арифметики был x87 FPU:
* 8 регистров, организованных как стек;
* расширенная точность;
* инструкции вроде `fld`, `fstp`, `fadd`.

На x86-64 обычный путь — SSE-регистры:
* `xmm0`, `xmm1`, ...;
* каждый `xmm` имеет размер 128 бит;
* в одном `xmm` можно хранить два `double` или четыре `float`;
* интерпретация битов зависит от инструкции.

Мы дальше будем использовать Intel-синтаксис ассемблера

## Вещественные аргументы в x86-64

В System V ABI:
* первые вещественные аргументы передаются в `xmm0`, `xmm1`, ...;
* вещественный результат возвращается в `xmm0`;
* `xmm`-регистры являются caller-saved.

Пример C-функции:

```c
double f(double x, double y) {
    return x * x + y;
}
```

Реализация на ассемблере:

```x86asm
    .intel_syntax noprefix
    .text
    .global f
f:
    mulsd xmm0, xmm0     // xmm0 = x * x
    addsd xmm0, xmm1     // xmm0 = x * x + y
    ret
```

## Скалярные SSE-инструкции

Скалярная инструкция использует только младшую часть `xmm`-регистра:
* `sd` — scalar double, младшие 64 бита;
* `ss` — scalar single, младшие 32 бита.

Частые инструкции:

```x86asm
movsd xmm0, qword ptr [rdi]  // загрузить double
movsd qword ptr [rdi], xmm0  // сохранить double
addsd xmm0, xmm1             // xmm0 += xmm1
subsd xmm0, xmm1             // xmm0 -= xmm1
mulsd xmm0, xmm1             // xmm0 *= xmm1
divsd xmm0, xmm1             // xmm0 /= xmm1
sqrtsd xmm0, xmm0            // xmm0 = sqrt(xmm0)
```

Аналогичные инструкции для `float`:

```x86asm
movss
addss
subss
mulss
divss
sqrtss
```

| Инструкция | Latency | Throughput | Комментарий |
|---|---:|---:|---|
| `movsd xmm0, [rdi]` | ~4 такта | ~0.5 такта | если данные уже в L1 cache |
| `movsd [rdi], xmm0` | нет обычной latency | ~1 такт | store уходит в store buffer |
| `addsd xmm0, xmm1` | ~4 такта | ~0.5 такта | можно начать до 2 независимых `addsd` за такт |
| `subsd xmm0, xmm1` | ~4 такта | ~0.5 такта | примерно как `addsd` |
| `mulsd xmm0, xmm1` | ~4 такта | ~0.5 такта | умножение не сильно дороже сложения |
| `divsd xmm0, xmm1` | ~13 тактов | ~4 такта | деление сильно медленнее |
| `sqrtsd xmm0, xmm0` | ~13 тактов | ~4.5 такта | корень тоже сильно медленнее |

### Пример: `x * x + y * y`

```c
double hyp2(double x, double y);
```

```x86asm
    .intel_syntax noprefix
    .text
    .global hyp2
hyp2:
    mulsd xmm0, xmm0
    mulsd xmm1, xmm1
    addsd xmm0, xmm1
    ret
```

Проверка:

```c
#include <stdio.h>

double hyp2(double x, double y);

int main() {
    printf("%f\n", hyp2(3.0, 4.0));
}
```

**OUTPUT:**
```
25.000000
```

## Packed SSE-инструкции

Packed-инструкция обрабатывает сразу несколько значений в одном `xmm`:
* `pd` — packed double, два `double`;
* `ps` — packed single, четыре `float`.

Пример для `double`:

```x86asm
movupd xmm0, xmmword ptr [rdi]  // загрузить два double
movupd xmm1, xmmword ptr [rsi]
mulpd xmm0, xmm1                // два умножения сразу
addpd xmm0, xmm1                // два сложения сразу
```

Пример для `float`:

```x86asm
movups xmm0, xmmword ptr [rdi]  // загрузить четыре float
movups xmm1, xmmword ptr [rsi]
mulps xmm0, xmm1                // четыре умножения сразу
addps xmm0, xmm1                // четыре сложения сразу
```

`movapd`/`movaps` требуют выравнивания адреса на 16 байт.
`movupd`/`movups` работают с невыровненными адресами.

## Горизонтальные операции

Обычные packed-инструкции работают "вертикально":

```
xmm0 = { a0, a1 }
xmm1 = { b0, b1 }

addpd xmm0, xmm1

xmm0 = { a0 + b0, a1 + b1 }
```

Инструкция `haddpd` складывает соседние элементы внутри регистров:

```
xmm0 = { x0, x1 }

haddpd xmm0, xmm0

xmm0 = { x0 + x1, x0 + x1 }
```

`haddpd` и `haddps` появились в SSE3, поэтому для таких примеров нужен флаг:

```sh
$ gcc -g -O0 -msse3 main.c funcs.S -o main
```

## Пример: скалярный `dot2`

```c
double dot2(const double *a, const double *b);
```

Функция считает:

```
a[0] * b[0] + a[1] * b[1]
```

Скалярная версия:

```x86asm
    .intel_syntax noprefix
    .text
    .global dot2
dot2:
    movsd xmm0, qword ptr [rdi]
    movsd xmm1, qword ptr [rsi]
    mulsd xmm0, xmm1

    movsd xmm2, qword ptr [rdi + 8]
    movsd xmm3, qword ptr [rsi + 8]
    mulsd xmm2, xmm3

    addsd xmm0, xmm2
    ret
```

## Пример: packed `dot2`

Та же функция через packed-инструкции:

```x86asm
    .intel_syntax noprefix
    .text
    .global dot2_packed
dot2_packed:
    movupd xmm0, xmmword ptr [rdi]   // { a[0], a[1] }
    movupd xmm1, xmmword ptr [rsi]   // { b[0], b[1] }
    mulpd xmm0, xmm1                 // { a[0]*b[0], a[1]*b[1] }
    haddpd xmm0, xmm0
    ret
```

Проверка:

```c
#include <stdio.h>

double dot2(const double *a, const double *b);
double dot2_packed(const double *a, const double *b);

int main() {
    double a[] = { 3.0, 4.0 };
    double b[] = { 10.0, 20.0 };

    printf("%f\n", dot2(a, b));
    printf("%f\n", dot2_packed(a, b));
}
```

**OUTPUT:**
```
110.000000
110.000000
```

## Пример: `dot4` для float

```c
float dot4(const float *a, const float *b);
```

Считаем:

```
a[0] * b[0] + a[1] * b[1] + a[2] * b[2] + a[3] * b[3]
```

```x86asm
    .intel_syntax noprefix
    .text
    .global dot4
dot4:
    movups xmm0, xmmword ptr [rdi]
    movups xmm1, xmmword ptr [rsi]
    mulps xmm0, xmm1
    haddps xmm0, xmm0
    haddps xmm0, xmm0
    ret
```

Проверка:

```c
#include <stdio.h>

float dot4(const float *a, const float *b);

int main() {
    float a[] = { 1.0f, 2.0f, 3.0f, 4.0f };
    float b[] = { 10.0f, 20.0f, 30.0f, 40.0f };

    printf("%f\n", dot4(a, b));
}
```

**OUTPUT:**
```
300.000000
```

| Инструкция | Latency | Throughput | Комментарий |
|---|---:|---:|---|
| `movups xmm0, xmmword ptr [rdi]` | ~4 такта | ~0.5 такта | загрузка 16 байт из L1 cache |
| `movups xmm1, xmmword ptr [rsi]` | ~4 такта | ~0.5 такта | вторая независимая загрузка из L1 cache |
| `mulps xmm0, xmm1` | ~4 такта | ~0.5 такта | четыре умножения `float` сразу |
| `haddps xmm0, xmm0` | ~6 тактов | ~2 такта | горизонтально складывает пары элементов |
| `haddps xmm0, xmm0` | ~6 тактов | ~2 такта | доводит сумму четырёх элементов до одного значения |
| `ret` | ~1-неск. тактов | зависит | обычно быстро, если return address предсказан |

## Сравнение вещественных чисел в SSE

Скалярные сравнения:

```x86asm
comisd xmm0, xmm1  // сравнить double: xmm0 ? xmm1
comiss xmm0, xmm1  // сравнить float:  xmm0 ? xmm1
```

После `comisd xmm0, xmm1` можно использовать условные переходы:
* `ja` — `xmm0 > xmm1`;
* `jb` — `xmm0 < xmm1`;
* `je` — `xmm0 == xmm1`;
* `jp` — unordered comparison, то есть встретился NaN.

Пример:

```c
int is_strictly_positive(double x);
```

Функция возвращает 1, если `x > 0.0`, и 0 иначе. Для NaN тоже возвращает 0.

```x86asm
    .intel_syntax noprefix
    .text
    .global is_strictly_positive
is_strictly_positive:
    xorpd xmm1, xmm1       // xmm1 = +0.0
    comisd xmm0, xmm1
    jp .false              // NaN
    ja .true               // x > 0.0
.false:
    xor eax, eax
    ret
.true:
    mov eax, 1
    ret
```

## Что важно запомнить

* `float` и `double` — конечные множества чисел, а не настоящие вещественные числа.
* `NaN` не равен ничему, включая самого себя.
* У нуля есть знак.
* Округление ломает привычную ассоциативность.
* На x86-64 вещественные аргументы и результат передаются через `xmm`-регистры.
* Скалярные SSE-инструкции работают с одним числом в младшей части `xmm`.
* Packed SSE-инструкции выполняют одну операцию сразу над несколькими числами.
* Векторизация может менять порядок операций, а значит и численный результат.
